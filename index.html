Project Brief & Flags

The Critical 5 project is structured as a set of cybersecurity “escape rooms”. Each room hides a short secret flag string that demonstrates the underlying vulnerability has been understood and exploited. My aim was to recover the flags, analyse why the weaknesses exist, and design practical mitigation and validation plans.

Rooms attempted

  • Task 1 IoT Temperature Data Access: accessed live sensor data and device controls using credentials obtained via social engineering.
  • Task 2 Fignal Network Traffic Analysis: captured HTTP traffic, extracted credentials and message contents, and proposed secure redesign.
  • Task 4 Membership Inference Attack: implemented a shadow_model attack against a ResNet18 classifier on CIFAR-10.

Key outcomes

  • Demonstrated end-to-end attack chains for each room (from discovery to exploitation and remediation).
  • Recovered all room flags and combined them into the final secret message (documented in the PDF portfolio).
  • Produced mitigation and validation plans that follow defence in depth and privacy by design principles.

Task 1 - IoT Temperature Data Access

IoT Social engineering HTTP APIs

Goal & threat model

Using Level 2 credentials discovered via password pattern analysis, I connected to the lab IoT house network and accessed the environmental control API. The attacker is assumed to have WiFi access and knowledge of default credentials but no physical access to the devices.

Approach

  • Joined the GL-SFT1200-253 network and verified local routing to the IoT gateway.
  • Used curl with HTTP headers for Username and Password to read temperature, PM2.5, humidity and VOC index.
  • Injected spoofed sensor values to trigger automated fan and LED responses, then issued direct commands to fan and LED topics for full manual control.

Key vulnerabilities

  • Static default credentials (tom/jerry) with full control permissions.
  • No role‑based access control or separation between read and write operations.
  • No server‑side validation of sensor ranges; arbitrary values trigger automations.
  • Plain HTTP traffic with no TLS and no audit logging of API activity.

Mitigation highlights

  • Introduce strong authentication with token‑based sessions and MFA for administrators.
  • Implement RBAC (Monitor / Operator / Admin) and strict sensor‑value validation rules.
  • Segment the network into sensor, control and management VLANs and enforce HTTPS with TLS.
  • Add anomaly detection and comprehensive audit logging for all API calls.

Task 2 – Fignal Network Traffic Analysis

Packet capture Wireshark Web security

Goal & approach

The Fignal room simulates an internal messaging service. By capturing loopback traffic with Wireshark while the client and server communicate, I identified the server, extracted user credentials, and analysed message contents.

  • Discovered the service at 192.168.100.72:5839 and confirmed all traffic was HTTP JSON.
  • Followed HTTP streams for /api/login to recover eight users’ full names, usernames, passwords and departments, plus their client IP addresses.
  • Reconstructed message flows for /api/messages, including security alerts and discussions of authentication bugs, all in plaintext.

Security findings

  • Credentials and messages transmitted in clear text, no HTTPS/TLS.
  • No rate‑limiting, account lockout or CAPTCHA to slow brute‑force attacks.
  • Predictable sequential user IDs and lack of message authentication, enabling spoofing.
  • Potential for session hijacking if cookies or tokens were used over HTTP.

Mitigation highlights

  • Deploy HTTPS with strong TLS and disable plain HTTP access to the API.
  • Hash passwords and replace repeated password transmission with signed JWT tokens.
  • Implement end‑to‑end encryption for particularly sensitive messages.
  • Enforce firewall rules restricting server access to authorised IP ranges only.

Task 4 – Membership Inference Attack

Machine learning Privacy ResNet18

Objective

This room demonstrates privacy leakage in ML. Using a ResNet18 trained on CIFAR-10, the aim was to infer whether a sample was in the model’s training set by looking at its prediction behaviour (member vs non-member).

Methodology

I trained three shadow models on disjoint CIFAR-10 subsets to approximate how the target behaves, then built an attack dataset from their prediction outputs. From each prediction I extracted confidence-style features (e.g., max probability, entropy, and confidence gap) and trained an attack classifier. Finally, I ran evaluate.py to test whether the trained attack model could generalise to the unseen target model.

Findings & implications

A key early lesson was that the baseline “83.33% accuracy” was misleading due to a 5:1 IN:OUT imbalance (the attack could succeed by always predicting IN). After balancing the data and using multiple shadows, the attack became meaningful and achieved strong IN/OUT separation, showing that overfitting and confidence differences can leak membership. In sensitive domains (healthcare/finance), this kind of leakage could expose whether someone’s data was used for training, raising GDPR-style privacy concerns.

Defence strategies

To reduce leakage, the target model should be less overfit (early stopping, weight decay, and dropout) so the train-test gap shrinks. Inference-time calibration like temperature scaling can also soften confidence/entropy signals that attacks rely on. For stronger protection, DP training (where feasible) and routine privacy testing should be part of a privacy-by-design workflow.
